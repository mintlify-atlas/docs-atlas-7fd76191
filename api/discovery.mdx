## Overview

Fast, persistent catalog of Jido components (Actions, Sensors, Agents, Plugins, Demos). Discovery uses `:persistent_term` for optimal read performance. The catalog is built asynchronously during application startup and can be refreshed on demand.

## Component Discovery

Discovery automatically finds and indexes:

- **Actions** - Discrete units of work (`__action_metadata__/0`)
- **Sensors** - Event monitoring components (`__sensor_metadata__/0`)
- **Agents** - Autonomous workers (`__agent_metadata__/0`)
- **Plugins** - Reusable capability packs (`__plugin_metadata__/0`)
- **Demos** - Example implementations (`__jido_demo__/0`)

## Component Metadata

Each discovered component includes:

```elixir
%{
  module: MyApp.CoolAction,
  name: "cool_action",
  description: "Does cool stuff",
  slug: "abc123de",
  category: :utility,
  tags: [:cool, :stuff]
}
```

## Functions

### init_async/0

Initializes the discovery catalog asynchronously. Call this from your application supervisor's start callback.

```elixir
Jido.Discovery.init_async()
```

### refresh/0

Refreshes the component catalog by rescanning all loaded applications.

```elixir
Jido.Discovery.refresh()
```

### last_updated/0

Returns the last time the catalog was updated.

```elixir
{:ok, timestamp} = Jido.Discovery.last_updated()
```

### catalog/0

Returns the full catalog for inspection.

```elixir
{:ok, catalog} = Jido.Discovery.catalog()
```

## Filtering Options

<ParamField path="limit" type="integer">
  Maximum results to return.
</ParamField>

<ParamField path="offset" type="integer">
  Results to skip (pagination).
</ParamField>

<ParamField path="name" type="string">
  Filter by name (partial match).
</ParamField>

<ParamField path="description" type="string">
  Filter by description (partial match).
</ParamField>

<ParamField path="category" type="atom">
  Filter by category (exact match).
</ParamField>

<ParamField path="tag" type="atom">
  Filter by tag (must have exact tag).
</ParamField>

Filters use AND logic - all specified filters must match.

## Usage Example

```elixir
defmodule MyApp.Application do
  use Application

  def start(_type, _args) do
    # Initialize discovery asynchronously
    _task = Jido.Discovery.init_async()

    children = [
      # Your supervision tree
    ]

    opts = [strategy: :one_for_one, name: MyApp.Supervisor]
    Supervisor.start_link(children, opts)
  end
end

# List AI-related actions
ai_actions = Jido.Discovery.list_actions(tag: :ai, limit: 20)

# Find a specific plugin
case Jido.Discovery.get_plugin_by_slug("abc123") do
  nil -> IO.puts("Plugin not found")
  plugin -> IO.inspect(plugin, label: "Found plugin")
end

# Pagination
page_1 = Jido.Discovery.list_actions(limit: 10, offset: 0)
page_2 = Jido.Discovery.list_actions(limit: 10, offset: 10)
```

## Implementing Discovery Metadata

### For Actions

```elixir
defmodule MyApp.CoolAction do
  use Jido.Action,
    name: "cool_action",
    description: "Does cool stuff",
    category: :utility,
    tags: [:cool, :stuff],
    schema: []

  def run(_params, _context) do
    {:ok, %{result: "cool"}}
  end
end
```

### For Plugins

```elixir
defmodule MyApp.MyPlugin do
  use Jido.Plugin,
    name: "my_plugin",
    description: "Provides X capability",
    category: :integration,
    tags: [:http, :api],
    state_key: :my_plugin,
    actions: [MyApp.PluginAction]
end
```

### For Sensors

```elixir
defmodule MyApp.MySensor do
  use Jido.Sensor,
    name: "my_sensor",
    description: "Monitors X metric"

  @impl true
  def init(config, _context) do
    {:ok, %{}}
  end

  @impl true
  def handle_event(event, state) do
    {:ok, state}
  end
end
```

## Performance

Reads are extremely fast (direct memory access) and never block. All processes can read concurrently without contention. The catalog is built once during startup and refreshed only when explicitly requested.
