## What is Jido?

Jido is a foundational framework for building autonomous, distributed agent systems in Elixir. It combines pure functional agent design with OTP's battle-tested concurrency primitives to give you deterministic, testable agents that run reliably in production.

**Core Philosophy:**
- Agents are immutable data structures
- State changes are pure data transformations
- Side effects are described as directives
- Runtime executes directives separately from agent logic

```elixir
{agent, directives} = MyAgent.cmd(agent, action)
```

## Quick Example

Build a simple counter agent in minutes:

<CodeGroup>

```elixir Agent Definition
defmodule CounterAgent do
  use Jido.Agent,
    name: "counter",
    description: "A simple counter agent",
    schema: [
      count: [type: :integer, default: 0]
    ],
    signal_routes: [
      {"increment", IncrementAction}
    ]
end
```

```elixir Action
defmodule IncrementAction do
  use Jido.Action,
    name: "increment",
    description: "Increments the counter",
    schema: [
      amount: [type: :integer, default: 1]
    ]

  def run(params, context) do
    current = context.state[:count] || 0
    {:ok, %{count: current + params.amount}}
  end
end
```

```elixir Usage
# Pure function usage (testable)
agent = CounterAgent.new()
{agent, directives} = CounterAgent.cmd(agent, {IncrementAction, %{amount: 5}})

agent.state.count  # => 5

# Runtime usage (production)
{:ok, pid} = MyApp.Jido.start_agent(CounterAgent, id: "counter-1")
{:ok, agent} = Jido.AgentServer.call(pid, Jido.Signal.new!("increment", %{amount: 10}))
```

</CodeGroup>

## Why Jido?

OTP primitives are excellent, but building multi-agent systems requires patterns that Jido formalizes:

| Raw OTP | Jido Formalizes |
|---------|-----------------|
| Ad-hoc message shapes per GenServer | Signals as standard envelope |
| Business logic mixed in callbacks | Actions as reusable command pattern |
| Implicit effects scattered in code | Directives as typed effect descriptions |
| Custom child tracking per server | Built-in parent/child hierarchy |
| Process exit = completion | State-based completion semantics |
