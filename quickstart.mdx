Get from zero to a working agent in 5 minutes. You'll learn how to define an agent, create actions, and run agents both as pure functions and with the OTP runtime.

## Prerequisites

- Elixir 1.18 or later
- Erlang/OTP 26 or later

<Steps>
  <Step title="Install Jido">
    Add Jido to your `mix.exs` dependencies:

    ```elixir mix.exs
    def deps do
      [
        {:jido, "~> 2.0"}
      ]
    end
    ```

    Then run:

    ```bash
    mix deps.get
    ```

    <Accordion title="Using Igniter (Recommended)">
      The fastest way to get started is with [Igniter](https://hex.pm/packages/igniter):

      ```bash
      mix igniter.install jido
      ```

      This automatically:
      - Adds Jido to your dependencies
      - Creates a `MyApp.Jido` instance module
      - Creates configuration in `config/config.exs`
      - Adds `MyApp.Jido` to your supervision tree

      See the [Installation](/installation) guide for complete manual setup instructions.
    </Accordion>
  </Step>

  <Step title="Define Your Agent">
    Agents are immutable structs with a schema defining their state:

    ```elixir lib/my_app/counter_agent.ex
    defmodule MyApp.CounterAgent do
      use Jido.Agent,
        name: "counter",
        description: "A simple counter agent",
        schema: [
          count: [type: :integer, default: 0],
          status: [type: :atom, default: :idle]
        ],
        signal_routes: [
          {"increment", MyApp.Actions.Increment}
        ]
    end
    ```

    Key points:
    - `schema` defines the agent's state with validation
    - `signal_routes` maps signal types to actions
    - Agents are just modules - no processes yet!
  </Step>

  <Step title="Define an Action">
    Actions transform agent state. They're pure functions that receive parameters and context:

    ```elixir lib/my_app/actions/increment.ex
    defmodule MyApp.Actions.Increment do
      use Jido.Action,
        name: "increment",
        description: "Increments the counter",
        schema: [
          amount: [type: :integer, default: 1]
        ]

      @impl true
      def run(params, context) do
        current = context.state[:count] || 0
        {:ok, %{count: current + params.amount}}
      end
    end
    ```

    The `run/2` function:
    - Receives validated `params` from the schema
    - Gets current state via `context.state`
    - Returns `{:ok, new_state}` or `{:ok, new_state, directives}`
  </Step>

  <Step title="Run Your Agent (Pure)">
    Test agents as pure functions without processes:

    ```elixir
    # Create an agent
    agent = MyApp.CounterAgent.new()

    # Execute an action
    {agent, directives} = MyApp.CounterAgent.cmd(agent, {MyApp.Actions.Increment, %{amount: 5}})

    # Check the result
    agent.state.count
    # => 5

    # Chain multiple actions
    {agent, _} = MyApp.CounterAgent.cmd(agent, {MyApp.Actions.Increment, %{amount: 3}})
    agent.state.count
    # => 8
    ```

    <Tip>
      Testing agents as pure functions is fast and doesn't require process setup. Use this for unit tests!
    </Tip>
  </Step>

  <Step title="Run with AgentServer (Production)">
    For production, run agents under supervision with `AgentServer`:

    ```elixir
    # Start the agent
    {:ok, pid} = MyApp.Jido.start_agent(MyApp.CounterAgent, id: "counter-1")

    # Send signals to the agent
    signal = Jido.Signal.new!("increment", %{amount: 10}, source: "/user")
    {:ok, agent} = Jido.AgentServer.call(pid, signal)

    agent.state.count
    # => 10

    # Look up by ID
    pid = MyApp.Jido.whereis("counter-1")

    # List all running agents
    agents = MyApp.Jido.list_agents()
    ```

    AgentServer provides:
    - Process-based execution
    - Signal routing
    - Directive execution
    - Parent-child hierarchies
    - Lifecycle management
  </Step>
</Steps>

## Common Patterns

<AccordionGroup>
  <Accordion title="How do I emit signals from an action?">
    Return directives from your action's `run/2`:

    ```elixir
    def run(params, context) do
      signal = Jido.Signal.new!("user.created", %{user_id: params.user_id})
      directive = Jido.Agent.Directive.emit(signal)

      {:ok, %{user_id: params.user_id}, [directive]}
    end
    ```
  </Accordion>

  <Accordion title="How do I spawn child agents?">
    Use the `SpawnAgent` directive:

    ```elixir
    def run(params, context) do
      directive = Jido.Agent.Directive.spawn_agent(
        ChildAgent,
        id: "child-1",
        tag: :worker,
        initial_state: %{task: params.task}
      )

      {:ok, %{spawned: true}, [directive]}
    end
    ```
  </Accordion>

  <Accordion title="How do I handle errors?">
    Return `{:error, reason}` from your action:

    ```elixir
    def run(params, context) do
      case validate(params) do
        :ok -> {:ok, %{validated: true}}
        {:error, reason} -> {:error, reason}
      end
    end
    ```

    Or use the `Error` directive for detailed error information:

    ```elixir
    error = Jido.Error.validation_error("Invalid input", details: %{field: :email})
    directive = Jido.Agent.Directive.error(error)

    {:ok, %{}, [directive]}
    ```
  </Accordion>

  <Accordion title="How do I schedule delayed work?">
    Use the `Schedule` directive:

    ```elixir
    def run(params, context) do
      signal = Jido.Signal.new!("reminder", %{message: params.message})
      directive = Jido.Agent.Directive.schedule(signal, 60_000) # 1 minute

      {:ok, %{scheduled: true}, [directive]}
    end
    ```
  </Accordion>
</AccordionGroup>

## Troubleshooting

<Warning>
  **Agent not receiving signals?** Make sure you've defined signal routes in your agent's `use Jido.Agent` declaration, or the signal type matches an action module name.
</Warning>

<Note>
  **State not updating?** Remember that `cmd/2` returns a *new* agent struct. You must use the returned agent, not the original.
</Note>
